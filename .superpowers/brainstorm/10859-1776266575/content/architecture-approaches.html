TradeMAX Rebuild — Architecture Approach

How should the Electron main process, IPC bridge, and React renderer be structured?

A

Monolithic Main Process + Thin Renderer

All business logic in main process. Services (trade engine, risk engine, AI, exchange APIs, auth, DB) live as modules in the main process. Renderer is purely a display layer — calls IPC for everything, receives event streams.

Pros

  • Maximum security — renderer has zero access to secrets
  • Simple mental model: main = brain, renderer = screen
  • Matches your "pure IPC rendering" requirement exactly
  • Single event loop for trade execution — no race conditions

Cons

  • Heavy main process — CPU-intensive work could block window management
  • Larger IPC surface area (~30+ channels)
  • Harder to unit test without Electron harness

⭐ Recommended — Cleanest security boundary. Trading systems need deterministic execution order, which a single-process pipeline guarantees.

B

Main Process + Worker Threads

Core orchestration in main, heavy computation in workers. Trade engine and AI service run in Node.js Worker Threads. Main process routes IPC and manages lifecycle. Renderer still thin.

Pros

  • Main process stays responsive — workers handle indicator math + AI calls
  • Can parallelize market analysis across symbols
  • Still secure — workers run in main context, not renderer

Cons

  • Significantly more complex — message passing between main ↔ workers
  • Shared state management becomes harder
  • Overkill for single-symbol trading with 8-second loop intervals
  • Harder to debug — stack traces cross thread boundaries
C

Microservice-Style (Main + Sidecar Processes)

Separate Node processes for each service domain. Main process spawns child processes for trade engine, exchange connectors, and AI service. They communicate via IPC pipes or local sockets. Renderer talks only to main.

Pros

  • Process isolation — crash in AI service doesn't kill trading
  • Can restart individual services independently
  • Scales to multi-exchange easily later

Cons

  • Massive complexity overhead for a desktop app
  • Multiple processes = more memory, harder packaging
  • IPC between processes adds latency to trade execution
  • Over-engineered for single-user, single-exchange scope
